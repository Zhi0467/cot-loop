\section*{Executive Summary}
\begin{itemize}[leftmargin=1.5em]
\item This is the larger follow-up to the earlier bounded OLMo2 \texttt{1B} fallback ladder. Wangzhi rejected the original \texttt{8}-prompt slices as too small to support stage conclusions, so this rerun scales the same object to \texttt{50} prompts per dataset.
\item The main base-versus-later conclusion survives: the large loop / max-length mass is already present in base, so the current evidence does not support ``degeneracy is introduced by SFT or RLVR.''
\item The bigger slice also changes the later-stage read. The earlier pilot understated the residual degeneration mass after SFT and final instruct. The later stages are not uniformly clean or monotone.
\item In the finished rerun, \texttt{RLVR1} is usually the cleanest point in the ladder. Final instruct is still much cleaner than base, but it can re-accumulate degeneration on some datasets, especially \texttt{GPQA} and \texttt{MMLU-Pro}.
\end{itemize}

\section*{Exact Object}
\begin{itemize}[leftmargin=1.5em]
\item Model ladder: base \texttt{->} SFT \texttt{->} \texttt{RLVR1} \texttt{->} instruct.
\item Datasets: \texttt{MATH-500}, \texttt{AIME}, \texttt{GPQA}, \texttt{MMLU-Pro}, and \texttt{LiveCodeBench release\_v6}.
\item Shared decode contract:
  \begin{itemize}[leftmargin=1.5em]
  \item \texttt{temperature=0.1}
  \item \texttt{top\_p=0.95}
  \item \texttt{top\_k=-1}
  \item \texttt{num\_generations=10}
  \item \texttt{max\_tokens=max\_model\_len=4096}
  \item \texttt{max\_num\_seqs=10}
  \item \texttt{max\_num\_batched\_tokens=2048}
  \item \texttt{max\_samples=50}
  \end{itemize}
\item Prompt surfaces:
  \begin{itemize}[leftmargin=1.5em]
  \item base math / MCQ: native \texttt{raw}
  \item SFT / \texttt{RLVR1} / instruct math / MCQ: native \texttt{chat\_template}
  \item \texttt{LiveCodeBench}: raw benchmark strings for every stage, with instruct checkpoints using the explicit \texttt{HFChatTemplate} LM style internally
  \end{itemize}
\item Output root bundle under the standard scratch tree:
  \begin{itemize}[leftmargin=1.5em]
  \item \texttt{olmo2\_1b\_degeneration\_origin\_progression/}
  \item \texttt{bound50\_temp0p1\_gen10\_ctx4096\_topkneg1/}
  \end{itemize}
\item The first replay attempt is intentionally excluded from this report because a write-path bug in \path{scripts/collect_model_stats.py} killed the jobs after generation and before JSON writeout. The table below is only the patched replay.
\item ``Correct'' below means rollout-level \texttt{num\_correct}, so each row is out of \texttt{500} generated rollouts rather than \texttt{50} prompts. For \texttt{LiveCodeBench}, native \texttt{pass@k} is reported separately.
\end{itemize}

\section*{What The Fifty-Prompt Rerun Changed}
\begin{itemize}[leftmargin=1.5em]
\item The \texttt{8}-prompt pilot was directionally right on the coarse question: base is still the main degeneration source.
\item The larger slice rules out the simpler later-stage story. The completed \texttt{AIME}, \texttt{GPQA}, and \texttt{MMLU-Pro} rows show meaningful residual degeneration after SFT and even after final instruct.
\item The cleanest later-stage point is usually \texttt{RLVR1}, not automatically the final instruct checkpoint.
\item So the earlier OLMo2 \texttt{8}-prompt table should now be treated as pilot evidence, not the stage conclusion itself.
\end{itemize}

\section*{Current Read}
\begin{itemize}[leftmargin=1.5em]
\item Base is still the dominant source of degeneration mass on this object.
\item Base is heavily degenerate on all four completed non-code datasets, not just on one cherry-picked slice.
\item SFT does not look like the origin of the behavior. Relative to base, it sharply reduces loop / max-length mass, but it still leaves substantial residual degeneration on some datasets.
\item \texttt{RLVR1} is the cleanest later-stage checkpoint on most datasets in the finished rerun.
\item Final instruct is mixed rather than monotone:
  \begin{itemize}[leftmargin=1.5em]
  \item much cleaner than base
  \item often cleaner than SFT
  \item not uniformly cleaner than \texttt{RLVR1}
  \item on \texttt{MMLU-Pro}, much worse than both SFT and \texttt{RLVR1}
  \end{itemize}
\item \texttt{LiveCodeBench} makes the base-origin split especially stark: base hits max length on 475 / 500 rollouts there, while SFT drops to 48 / 500, \texttt{RLVR1} to 4 / 500, and instruct to 2 / 500.
\item Native coding success stays weak across the whole \texttt{1B} ladder: \texttt{pass@10 = 0.0} for base, SFT, and instruct, and only \texttt{0.02} for \texttt{RLVR1}. So this benchmark is more useful here as a degeneration surface than as a capability-ranking surface.
\item So the larger rerun keeps the main base-origin read but rules out the simpler ``all later stages are basically clean'' interpretation from the pilot.
\end{itemize}

\section*{What This Rerun Still Does Not Prove}
\begin{itemize}[leftmargin=1.5em]
\item This is still the cheap OLMo2 fallback ladder, not a same-family OLMo3 \texttt{base -> SFT -> RLVR} result.
\item \texttt{LiveCodeBench} capability is weak across the whole \texttt{1B} ladder, so that row should be read mainly for loop / max-length behavior rather than as a benchmark-capability claim.
\end{itemize}
