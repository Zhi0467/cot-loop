\section*{Executive Summary}
\begin{itemize}[leftmargin=1.5em]
\item The first bounded OLMo3 read really was fishy. Two concrete evaluation bugs were mixed into it:
  \begin{itemize}[leftmargin=1.5em]
  \item \texttt{RLVR / MMLU-Pro = 0 / 80} came from rejecting relaxed terminal forms like \verb|{"answer": I}|.
  \item the first OLMo \texttt{LiveCodeBench} rows used a silent Qwen-specific wrapper fallback instead of an OLMo-native prompt path.
  \end{itemize}
\item Those rows are now repaired on the saved outputs. The corrected bounded OLMo3 instruct bundle does \textbf{not} support ``RLVR introduces the degeneracy.''
\item The OLMo3 base stage is still only available on the corrected \texttt{32}-prompt \texttt{MATH-500} pilot, because the full \texttt{7B} base sweep was too slow to finish honestly under the bounded multi-dataset contract.
\item The cheap full-ladder control is now the bounded OLMo2 \texttt{1B} progression, and that one shows the stage shape clearly: heavy loop / max-length mass in base, reduced but still present mass in SFT, and much smaller mass in \texttt{RLVR1} / instruct.
\item Low accuracy and degeneracy are not the same thing on these objects. Some later-stage rows stay weak in accuracy while already having little or no loop / max-length mass.
\end{itemize}

\section*{What Was Repaired}

\subsection*{MMLU-Pro}
The earlier \texttt{RLVR / MMLU-Pro = 0 / 80} row was not a real zero-capability collapse. Saved completions often ended with relaxed JSON-like forms such as \verb|{"answer": I}|, and the old parser rejected that exact form even when the answer letter was correct. After patching \path{src/loop_probe/adapters/_common.py}, the corrected bounded RLVR row is \texttt{27 / 80} correct with \texttt{0 / 80} loops and \texttt{0 / 80} max-length hits.

\subsection*{LiveCodeBench}
The first OLMo \texttt{LiveCodeBench} rows were also on the wrong surface. The adapter had been silently defaulting non-Qwen families to a Qwen-specific LM style. \path{src/loop_probe/adapters/livecodebench_codegen.py} now has an OLMo-native path: instruct checkpoints serialize the benchmark prompt through the model tokenizer chat template into a final raw string, raw/base checkpoints stay on the generic raw path, and code extraction now tries fenced blocks first before raw-code fallback.

\section*{Current Conclusion}
\begin{itemize}[leftmargin=1.5em]
\item The fishy OLMo3 rows were not a vague ``maybe wrong'' concern. They came from two concrete evaluation-surface bugs, and the corrected rows are now on disk.
\item The corrected bounded OLMo3 evidence does not support ``degeneration is introduced during RLVR.''
\item The bounded OLMo2 \texttt{1B} ladder already shows the stage progression the original note was trying to isolate: heavy degeneracy in base, reduced but still present mass in SFT, and much smaller mass in \texttt{RLVR1} / instruct.
\item What is still missing is a full OLMo3 \texttt{7B} base \texttt{->} SFT \texttt{->} RLVR five-dataset bundle. The blocker there is runtime, not remaining interface ambiguity.
\end{itemize}
