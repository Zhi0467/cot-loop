\section*{One correction first}
The previous version used the wrong selector. The \texttt{top 20\%} bucket test
answers a downstream question: after training, which score happens to
concentrate loop-heavy prompts best? Wangzhi's actual question is different:
which prompt-level label or regression target can we predict well from
prompt-prefill activations on held-out prompts? Those are not the same
question.

\section*{One-sentence answer}
If the selector is held-out predictability rather than downstream bucket
behavior:
\begin{itemize}[leftmargin=1.5em]
\item best regression target right now: \texttt{mean\_relative\_length}
\item best binary label right now: \texttt{majority\_s\_0.5}
\item \texttt{p\_loop} is still the most loop-specific target, but the current
saved runs do not show that it is the best-predicted target
\end{itemize}

\section*{What the project is actually trying to do}
Fix one \texttt{(model, decode policy)} pair. For each prompt, define one
prompt-level label or regression target from repeated rollouts. Train on
prompt-prefill activations only. Then choose the target that the current probes
can predict well on held-out prompts. Only after that does it make sense to ask
whether the chosen target is useful for a downstream screen.

\section*{Definitions}
\textbf{\texttt{p\_loop}}: for one prompt, what fraction of repeated rollouts
enter the loop regime.

\textbf{\texttt{p\_cap}}: for one prompt, what fraction of repeated rollouts hit
the hard max-length cap.

\textbf{\texttt{mean\_relative\_length}}: for one prompt, what fraction of the
available generation budget gets used on average. This is a dense continuous
target.

\textbf{\texttt{majority\_s\_0.5}}: a binary label that is \texttt{1} if more
than half of repeated rollouts use at least half of the available budget. This
is better read as ``often long'' than as ``often loops.''

\textbf{\texttt{effective\_budget}}: the real generation budget left after
prompt length and context limits are taken into account.

\textbf{Flagged top-risk bucket}: only an old evaluation slice. For one trained
score on held-out prompts, take the \texttt{20\%} with the largest predicted
score and inspect what actually happened there. It is not the project goal and
not the target-selection rule.

\textbf{Loop-rate enrichment}: loop rate inside the flagged bucket divided by
overall loop rate on the same test split.

\section*{What was trained}
This pass used the saved prompt-profile bundles for \texttt{GPQA},
\texttt{AIME}, \texttt{MATH-500}, \texttt{MMLU-Pro}, and
\texttt{LiveCodeBench}. The saved surface already contained activation probes
for \texttt{majority\_s\_0.5}, \texttt{p\_loop}, and
\texttt{mean\_relative\_length}, each with a last-layer MLP and a per-layer
ensemble MLP.

This pass also added the missing metadata-only controls:
\begin{itemize}[leftmargin=1.5em]
\item logistic regression for \texttt{p\_loop}
\item linear regression for \texttt{mean\_relative\_length}
\item features = \texttt{prompt\_length}, \texttt{effective\_budget}, and both
together
\end{itemize}

The split stayed prompt-disjoint: train prompts and test prompts are different
prompts, and repeated rollouts from one prompt never leak across the split.

\section*{What should pick the target}
The selector should be held-out fit on the target itself.
\begin{itemize}[leftmargin=1.5em]
\item For a binary label: held-out classification quality on that label itself.
\item For a continuous target: held-out regression quality on that target
itself.
\end{itemize}

The bucket test is a downstream diagnostic. It should not choose the training
target.

\section*{What this means for \texttt{p\_loop}}
\texttt{p\_loop} is still important, but the claim has to be narrower.
\begin{itemize}[leftmargin=1.5em]
\item It is the most direct loop-frequency target in this bundle.
\item The old bucket test still shows that a \texttt{p\_loop} score can isolate
loop-heavy prompts downstream.
\item But the current saved runs do not show that \texttt{p\_loop} is the
target we can predict best.
\end{itemize}

The most obvious difference in the saved tables is target density:
\begin{itemize}[leftmargin=1.5em]
\item held-out \texttt{p\_loop} target means are only \texttt{0.104}
(\texttt{AIME}), \texttt{0.138} (\texttt{GPQA}), \texttt{0.025}
(\texttt{MATH-500}), \texttt{0.036} (\texttt{MMLU-Pro}), and \texttt{0.148}
(\texttt{LiveCodeBench})
\item \texttt{mean\_relative\_length} is dense and uses every rollout
\end{itemize}

That density explanation is an inference from the saved results, not a new
experiment.

\section*{So what is the right objective?}
There are two honest answers, depending on output type:
\begin{itemize}[leftmargin=1.5em]
\item If the project wants one continuous target to regress well from prefill
activations, use \texttt{mean\_relative\_length}.
\item If the project wants one binary label to classify with the current saved
surface, use \texttt{majority\_s\_0.5}.
\end{itemize}

What I should not say anymore is ``train \texttt{p\_loop} because it wins
loop-rate enrichment.'' That used the wrong rule.

\section*{What this resolves, and what it does not}
This fixes the main logic error in the previous write-up. The saved bundle now
supports a clean proved-versus-unproved ledger.

Proved on the current saved bundle:
\begin{itemize}[leftmargin=1.5em]
\item \texttt{mean\_relative\_length} is the strongest regression target by
held-out fit
\item \texttt{majority\_s\_0.5} is the strongest existing binary label surface
\item \texttt{p\_loop} still contains real activation signal beyond metadata
\end{itemize}

Not proved on the current saved bundle:
\begin{itemize}[leftmargin=1.5em]
\item that \texttt{p\_loop} should be the main training objective
\item that the project should collapse binary and regression into one final
answer without first choosing output type
\item that the old bucket result should select the label
\end{itemize}

Three things still remain open:
\begin{enumerate}[leftmargin=1.5em]
\item this is retrospective on saved bundles rather than one fresh prospective
run
\item \texttt{LiveCodeBench} still lacks prompt-level accuracy in the recovered
projection artifact
\item if the real goal is specifically loop prediction, then the current answer
is not ``use \texttt{p\_loop}''; it is ``we have not yet shown that
\texttt{p\_loop} is the most learnable target''
\end{enumerate}

\section*{Exact next run}
\begin{itemize}[leftmargin=1.5em]
\item Decide the output type first.
\item If regression is the product, compare \texttt{mean\_relative\_length} and
\texttt{p\_loop} prospectively and choose by held-out regression quality plus
metadata lift.
\item If binary classification is the product, keep \texttt{majority\_s\_0.5}
as the current binary control and only reopen a direct loop-derived binary label
if it actually wins on held-out classification quality.
\item If the goal is specifically to rescue \texttt{p\_loop}, judge it by
held-out \texttt{p\_loop} fit, not by bucket enrichment.
\end{itemize}
